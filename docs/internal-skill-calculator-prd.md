# 内功计算器页面级 PRD（内部原型）

> 状态：内部评审稿，不面向公开发布。规则来源为
> `7.8新世界内功计算器(9)(8).xlsx`。Excel 只用于规则核对与回归测试，不作为运行时依赖，
> 不随应用读取、解析或分发。

## 1. 产品背景

现有软件主要解决宏流程的执行效率，内功计算器用于补充配装决策能力。原 Excel 已能给出
词条分、特性分、总分和档位，但输入分散、联动关系不透明，也不适合直接嵌入桌面应用。

本原型将表格规则重构为原生页面，让用户完成三件事：快速录入、理解分数来源、判断距离
下一档还差多少。原型验证规则正确性和页面可用性，不代表最终商业发布方案。

## 2. 目标与非目标

### 2.1 目标

- 完整承载 11 个基础属性、15 个内功的携带与“灵”状态、1 个周天选择。
- 实时输出词条分、特性分、总分、当前档位和下一档差值。
- 展示逐项贡献及灼星贯日的联动来源，使结果可解释、可复核。
- 使用版本化、确定性的应用内规则；同一输入始终得到同一结果。
- 以原表默认数据作为黄金样本，建立后续实现和回归测试基线。

### 2.2 非目标

- 不在 MVP 中提供截图 OCR、自动识别、最优组合搜索或游戏内自动操作。
- 不把 Excel 界面、单元格公式引擎或 Office/COM 能力搬入应用。
- 不从用户磁盘读取原工作簿，不依赖网络表格或远程规则服务。
- 不在规则约束未确认前强制“最多携带 N 个”或推断“灵”与携带的关系。
- 不宣称计算结果为官方标准，也不据此提供交易、付费或收益承诺。

## 3. 核心用户任务

1. 用户录入当前装备的基础属性。
2. 用户在同一内功行勾选“携带”和“灵”，并选择周天。
3. 用户立即看到总分、档位、分项贡献和距离下一档的差值。
4. 用户通过贡献明细和联动说明定位最有影响的属性或内功。
5. 用户默认从空白方案开始，可随时清空当前录入后重新计算。

## 4. 信息架构

将计算器作为独立一级工作区，不塞入现有宏流程侧栏或弹窗。

```text
应用
├─ 宏流程
└─ 内功评估
   └─ 当前计算（MVP）
      ├─ 基础属性
      ├─ 内功配置
      ├─ 周天
      └─ 结果与贡献明细
```

P1 再在“内功评估”下增加已保存方案和 A/B 对比。主题仅影响视觉，不得改变计算规则；
职业主题也不得自动替换规则集。

## 5. 页面结构

### 5.1 页头

- 标题：`内功评估`。
- 状态标识：`内部原型`。
- 只读规则标识：`7.8 新世界 · 防守团`，同时显示内部规则版本。
- 操作：`清空全部`。无输入时按钮禁用；存在录入时清空当前草稿。
- 来源入口：打开页面内的规则来源与免责声明，不跳转不明外链。

### 5.2 主体布局

默认窗口宽度下采用三栏：基础属性、内功配置、结果。结果栏在页面滚动时保持可见。

```text
┌ 基础属性 ──────┬ 内功配置 ─────────────────────┬ 结果 ────────────┐
│ 11 个数值输入  │ 内功名称  携带  灵  本项贡献   │ 总分 / 当前档位  │
│ 输入单位提示   │ 15 行 + 联动说明              │ 距下一档差值     │
│                │ 周天：金火 / 火木 / 金木      │ 词条分 / 特性分  │
│                │                               │ 贡献明细         │
└────────────────┴───────────────────────────────┴──────────────────┘
```

- 窗口空间不足时结果栏移至下方，不产生全局横向滚动。
- 内功区使用表格语义；“携带”和“灵”位于同一行，贡献同时展示词条与特性来源。
- 结果区优先级依次为总分、档位、下一档差值、两类分数、贡献明细。
- 分数主展示保留 2 位小数；展开明细保留 4 位小数。档位判断使用未四舍五入的原始总分。

## 7. 计算规则

### 7.1 总体公式

```text
词条分 = 11 项基础属性得分之和 + 15 项灵得分之和
特性分 = 所有已携带内功的特性分之和 + 周天加成
总分   = 词条分 + 特性分
```

计算过程使用完整精度，不对中间项逐项四舍五入。显示层舍入不得反向参与总分或档位计算。
规则应固化为应用内的版本化结构数据和确定性计算函数；Excel 不参与运行时计算。

### 7.2 灼星贯日联动

定义联动基数：

```text
linkedBase = 5.5
  +（众妙为灵 ? 7 : 众妙已携带 ? 5.25 : 0）
  +（绝电惊沙为灵 ? 8 : 绝电惊沙已携带 ? 5.5 : 0）
```

- 灼星贯日已携带时，特性分为 `3 + 0.2 × linkedBase`；未携带为 0。
- 灼星贯日为灵时，词条分为 `1.875 + 0.1 × linkedBase`；不为灵时为 0。
- 在众妙、绝电惊沙行以及灼星贯日行展示“影响灼星贯日”的联动标识。
- 用户变更任一关联项后，灼星贯日的两类贡献和总分同步更新。
- `灵` 优先于同一内功的普通携带值参与联动；即使同时勾选，也只采用灵分支一次。

### 7.3 档位边界

原表使用区间文案但未明确等号归属。内部原型暂按“下限包含、上限不包含”实现：

| 原始总分区间    | 档位       |
| --------------- | ---------- |
| `< 63`          | 建议转生   |
| `>= 63 且 < 66` | 小哥布林   |
| `>= 66 且 < 69` | 大哥布林   |
| `>= 69 且 < 72` | 哥布林精英 |
| `>= 72 且 < 75` | 哥布林将军 |
| `>= 75`         | 哥布林国王 |

边界使用原始总分判断。例如 `71.9999` 仍属于哥布林精英，显示成 `72.00` 也不能提前升档。
若来源方确认不同归属，应修改规则版本并补充迁移说明，不得静默改动旧版本结果。

“距下一档”只在存在下一档时显示，计算为下一档下限减原始总分；最高档显示“已达最高档”。

## 8. 交互与状态

### 8.1 实时计算

- 有效输入变化后立即本地计算，无需提交按钮、加载动画或网络请求。
- 数值框支持键盘录入、全选和清除；空数值按 0 参与计算。
- `清空全部`将数值、携带和灵清零，周天恢复金火；无草稿内容时按钮禁用。
- 改动后标记页面为“未保存”仅留作 P1；MVP 不暗示数据已持久化。
- 展开贡献明细时按“基础属性、灵、携带特性、周天”分组，默认隐藏 0 贡献项，可切换显示全部。

### 8.2 验证与错误

- 数值仅接受有限、非负十进制；不设置未经确认的业务上限，也不静默截断或修正。
- 负数、无穷值或无法解析的内容显示字段级错误；存在错误时总结果显示“请修正输入”，不输出新档位。
- 百分比字段始终显示 `%` 后缀和“填写百分号前数值”提示。
- 携带数仅显示 `已携带 N`，在最大数量未确认前不阻止继续选择。
- 携带与灵允许独立组合；MVP 不把看似异常的组合视为错误。
- 周天不可为空；程序收到未知值时按数据错误处理，不自动回退并掩盖问题。

### 8.3 空状态

首次进入或清空后，若所有数值为 0 且没有任何携带或灵选择，结果区显示：
`填写属性或选择内功后查看评估结果`。此时不展示“建议转生”，避免把尚未开始录入误解为真实
评估。只选择周天不视为有效方案。

### 8.4 可访问性

- 每个输入都有持久可见的文本标签，不以 placeholder 代替；错误提示与字段通过
  `aria-describedby` 关联。
- 内功使用有表头的表格语义；复选框可通过 Tab 到达，名称需包含内功名和动作，例如
  “携带灼星贯日”“灼星贯日为灵”。
- 周天使用具名 `fieldset` 和单选语义；所有操作支持键盘，不依赖悬停。
- 焦点环清晰，文字对比度至少 4.5:1，状态不能只用颜色表达。
- 总分摘要更新使用克制的 `aria-live="polite"`，连续输入时避免逐键重复朗读完整明细。
- 尊重 `prefers-reduced-motion`；主题装饰和纯净模式不得影响字段可读性或点击区域。

## 9. 版本范围

### MVP：内部原型

- 独立“内功评估”页面及本 PRD 全部字段。
- 实时计算、档位、下一档差值、携带计数、分组贡献明细。
- 灼星贯日联动解释。
- 空白首屏、清空、输入验证、空状态和来源说明。
- 版本化规则与黄金样本、边界、周天、联动的自动化测试。
- 不持久化方案，不导入/导出，不采集原始输入。

### P1：可日常使用

- Rust 侧持久化多个计算方案，统一通过受限 Tauri API 调用，不使用 `localStorage` 作为正式来源。
- A/B 方案对比、差异高亮、最近使用和 JSON 导入/导出。
- 生成本地结果图片；图片明确带规则版本与“非官方”标识。
- 正式版新建方案的默认值策略、携带数量规则和来源授权落定。

### P2：扩展能力

- 多赛季、多流派规则包与旧版本只读复算。
- 截图 OCR 与人工校对流程。
- 在携带数量及游戏约束确认后提供最优组合建议。
- 经签名、校验和回滚保护的规则更新机制；仍不以 Excel 作为运行时格式。

## 10. 埋点建议

内部原型默认不启用遥测。若后续产品已有用户同意的匿名遥测能力，只记录聚合行为，不记录
基础属性数值、内功组合、方案名称、截图或设备路径。

| 事件                           | 触发时机                 | 建议属性                         |
| ------------------------------ | ------------------------ | -------------------------------- |
| `calculator_viewed`            | 首次进入页面             | `ruleVersion`、`appVersion`      |
| `calculator_first_result`      | 本次访问首次产生有效结果 | `ruleVersion`、`tier`            |
| `calculator_cleared`           | 清空全部                 | `hadValidResult`                 |
| `calculator_validation_failed` | 出现输入错误             | `fieldId`、`errorType`，不含原值 |
| `calculator_detail_toggled`    | 展开或收起贡献明细       | `expanded`                       |

不要对每次键入上报事件；同一错误应在一次编辑会话内去重。

## 11. 验收标准

### 11.1 规则正确性

- 黄金样本得到 `38.1479 / 33.55 / 71.6979`，误差不超过 `0.000001`。
- 黄金样本显示“哥布林精英”和“距下一档 0.30”。
- 金火切换为火木时特性分和总分均准确增加 2.7；切换为金木时相对金火增加 2.8。
- 黄金样本中把绝电惊沙改为灵后，灼星贯日特性贡献增加 0.5、灼星贯日灵贡献增加
  0.25，绝电惊沙灵贡献增加 2.46，总分合计增加 3.21。
- 分别覆盖 `62.9999、63、66、69、72、75`，档位符合第 7.3 节临时边界定义。
- 显示舍入不改变档位；所有 0 贡献项不会产生负零或 `NaN`。

### 11.2 页面与交互

- 11 个基础字段、15 行携带与灵、3 个周天选项均可仅用键盘完成操作。
- 空白首屏、清空、错误状态和空状态与第 8 节一致。
- 每个非零贡献可追溯到具体字段、内功或周天，灼星贯日联动值可解释。
- 1080×700、1280×820、最大化，以及 100%、125%、150% DPI 下无全局横向滚动或控件遮挡。
- 所有内置主题和纯净模式下均可读；主题切换不改变任何分数。

### 11.3 技术边界

- 安装包不读取或打包该 Excel，不引入 Office/COM、通用文件系统或 Shell 权限。
- 规则有稳定版本 ID，计算层可在无 UI、无 Tauri、无网络环境下执行自动化测试。
- 方案持久化未进入 MVP；P1 实现时磁盘读写必须留在 Rust，并通过受限 API 暴露。
- 实现提交至少通过前端测试、`pnpm typecheck`、`pnpm build` 及项目规定的 Rust 检查。
